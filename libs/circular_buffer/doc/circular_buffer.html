描述

术语 循环缓冲区circular buffer 通常指的是内存中的一块用于保存输入数据的区域。当缓冲区被填入时，新的数据从缓冲区的开始被写入，并覆盖掉旧的数据。(见图)

circular_buffer 是一个符合STL的容器。它是一种类似于 std::list 或 std::deque 的序列。它支持随机访问迭代器，在缓冲区头部或尾部的常量时间插入和删除操作，与 std 算法的互操作能力。circular_buffer 被特别设计为提供固定容量的存储大小。当其容量被用完时，新插入的元素会覆盖缓冲区头部或尾部(取决于使用何种插入操作)的元素。

circular_buffer 仅在创建时、显式调整容量时，或需要重新调整大小或赋值操作时才进行内存分配。另一方面，还有一个 circular_buffer_space_optimized 可用。它是 circular_buffer 的一个适配器，它不在创建时分配内存，而是在需要时分配内存。

介绍性示例

以是一个使用 circular_buffer 的简单例子： 

   #include <boost/circular_buffer.hpp>

   int main(int /*argc*/, char* /*argv*/[]) {

      // 创建一个容量为3个整数的循环缓冲区。
      boost::circular_buffer<int> cb(3);

      // 插入一些元素到缓冲区。
      cb.push_back(1);
      cb.push_back(2);
      cb.push_back(3);

      int a = cb[0];  // a == 1
      int b = cb[1];  // b == 2
      int c = cb[2];  // c == 3

      // 现在缓冲已满，再插入元素将覆写最前面的元素。

      cb.push_back(4);  // 用 4 覆盖 1.
      cb.push_back(5);  // 用 5 覆盖 2.

      // 现在缓冲区包含 3, 4 和 5.

      a = cb[0];  // a == 3
      b = cb[1];  // b == 4
      c = cb[2];  // c == 5

      // 可以从前端或后端弹出元素。

      cb.pop_back();  // 5 被删除。
      cb.pop_front(); // 3 被删除。

      int d = cb[0];  // d == 4

      return 0;
   }

原理

circular_buffer 背后的动机是创建一个可以与STL无缝使用的容器。另外，circular_buffer 的设计受到以下原则的指导：

  1. 对于常见应用具有最大的效率。
  2. 适用于多种用途。
  3. 缓冲区的行为应尽可能符合直觉。
  4. 适合于通过适配器进行特化。(circular_buffer_space_optimized 就是一个适配器的例子)
  5. 易于调试。(详见 对调试的支持)

为了达到最高的效率，circular_buffer 将它的元素保存在一片连续内存中，这样可以：

  1. 使用固定的内存，不需要隐式或意外的内存分配。
  2. 从前端或后端进行快速的常量时间的插入和删除元素。
  3. 快速的常量时间的对元素进行随机访问。
  4. 适用于实时和对性能有严格要求的应用程序。

circular_buffer 的可能应用包括：

  • 保存最近接收到的取样数据，在新的取样数据到达时覆盖最旧的数据。
  • 作为有界缓冲区的底层容器(请见 有界缓冲区示例)。
  • 一种用于保存特定数量的最后插入元素的快速缓冲。
  • 高效的固定容量FIFO(先进先出)或LIFO(后进先出)队列，当队列满时删除最旧的(即最早插入的)元素。

circular_buffer 的：

线程安全性

circular_buffer 的线程安全性与大多数STL实现中的容器的线程安全性相同。即 circular_buffer 不是线程安全的。线程安全仅在以下情形时被保证：对 circular_buffer 的不同实例进行并发访问是安全的，对共享的 circular_buffer 进行并发读访问是安全的。

如果多个线程访问同一个 circular_buffer, 且至少有一个线程可能会进行写访问，则用户要负责确保各个线程在访问容器时互斥。线程间的互斥可以通过对底层的 circular_buffer 操作增加一个锁获取和释放来实现。(请见 有界缓冲区示例)

覆写操作

覆写操作发生在当一个元素被插入到一个满的 circular_buffer 中时 - 旧的元素将被新元素所覆盖。在正式审查时，曾经对"覆写某个元素"的精确意义进行过讨论。它可能是析构一个原有的元素并接着原地构造一个新元素，也可能是将新元素赋值到旧元素中。circular_buffer 最终实现了赋值操作，因为它更为高效。

从被保存元素的业务逻辑观点来看，析构/构造操作与赋值操作其实是一样的。但是，在极少见的情形下(如果有)它们之间可能有不同。如果元素本身需要 以析构/构造操作来替换赋值操作，可能考虑对该元素实现一个赋值操作的包装器，并改为保存该包装器。需要注意，保存这个包装器有一个缺点。对该包装器的每 次赋值都要调用析构/构造操作 - 不仅是在该包装器被覆写时(即缓冲区满时)，也会发生在对被保存包装器进行移位时(例如，在容器中央插入)。

写入到一个满的缓冲区

当数据源产生的数据多于固定大小缓冲区可以容纳的数量时，通常有以下几种选择：

  1. 通知数据源等待，直至缓冲区有空间为止(如通过抛出一个溢出异常)。
  2. 如果最旧的数据最为重要，则忽略来自数据源的新数据，直至缓冲区有空间为止。
  3. 如果最新的数据最为重要，则覆写最旧的数据。
  4. 让数据产生者负责在写入数据前检查缓冲区的大小。

显然，circular_buffer 实现了第三种选择。但是它不实现其它选择 - 尤其是前两个 - 则可能不太明显。你可能觉得 circular_buffer 应该实现前面三种选项并提供一个机制来在它们之间进行选择。这种印象是错的。circular_buffer 被设计并优化为循环的(这意味着在缓冲区满时将覆写最旧的数据)。如果有这样一种控制机制，它只会把事情弄复杂，而且 circular_buffer 的使用也可能会不够方便。

此外，前两个选项(还有第四个)并不要求缓冲区是循环的。如果需要前两个选项，请考虑实现 std::vector 的一个适配器。这种情况下，circular_buffer 不适合用于适配，因为和 std::vector 相比，它要为其循环行为负出一定的开销。

从一个空的缓冲区读取/删除

当从一个空缓冲区读取或删除元素时，缓冲区应可以通知数据消耗者(例如通过抛出一个下溢异常)它没有元素保存。因为以下两个原因，circular_buffer 没有实现这一行为：

  1. 它会引入性能的开销。
  2. 其它 std 容器也没有这样做。

从一个空的 std 容器中读取或删除一个元素(如通过调用 front() 或 pop_back())被认为是 bug，circular_buffer 也一样。数据消耗者在从容器中读取/删除元素之前，必须测试它是否为空。不过，从 circular_buffer 读取时，有一个选项决定 at() 方法在索引超界时是否抛出异常。

迭代器失效

如果一个迭代器所指的元素被删除或被另一个元素覆写，通常认为该迭代器将失效。该规则是 对调试的支持 所强制的，并且被记录在每一个方法中。不过，有些使用 circular_buffer 的应用可能要求较松的规则：迭代器仅当其指向未初始化内存时才无效。考虑以下例子： 

   #define BOOST_CB_DISABLE_DEBUG // 调试支持已关闭，否则代码将引发运行期错误。

   #include <boost/circular_buffer.hpp>
   #include <assert.h>

   int main(int /*argc*/, char* /*argv*/[]) {

      boost::circular_buffer<int> cb(3);

      cb.push_back(1);
      cb.push_back(2);
      cb.push_back(3);

      boost::circular_buffer<int>::iterator it = cb.begin();

      assert(*it == 1);

      cb.push_back(4);

      assert(*it == 4); // 该迭代器仍指向已初始化的内存。

      return 0;
   }

该迭代器不再指向原来的元素(从"严格"的观点来看，它被认为是无效的)，但它仍然指向内存中同一个有效的位置。 circular_buffer 所支持的对迭代器失效的"宽松"定义被认为是实现细节，而不是完整的特性。迭代器依然有效的规则可以由 soft_iterator_invalidation.cpp 中的代码推断得到。

警告

circular_buffer 不能用于保存指向动态分配对象的指针。当一个 circular_buffer 变满时，后来的插入将覆写已保存的指针 - 这会导致内存泄漏。建议采用智能指针[1]来代替。(任何 std::auto_ptr 容器都被认为是非常危险的[2])

虽然 circular_buffer 的内部是循环的，但它的迭代器不是。circular_buffer 的迭代器仅在范围 [begin(), end()] 内有效。如，迭代器 (begin() - 1) 和 (end() + 1) 均为无效。

对调试的支持

为了帮助程序员避免和发现常见的 bugs, circular_buffer 包括了一些对调试的支持。

circular_buffer 维护了一个有效迭代器列表。一旦任何元素被销毁，指向该元素的所有迭代器将被从这个列表中删除，明确失效(设置一个失效标志)。对调试的支持还包括多个断言(BOOST_ASSERT 宏)，这样可以确保 circular_buffer 和它的迭代器在运行期正确使用。当一个无效迭代器被使用时，断言将报告一个错误。显式的迭代器失效和断言的使用是一种非常强的调试技术，可以捕捉到大多数错误。

此外，在调试模式下，由 circular_buffer 分配的未初始化内存会用值 0xcc 进行填充。这样可以帮助程序员在调试代码时分辨出已初始化和未初始化的内存。详情请参见源代码。

对调试的支持仅在调试模式下打开(即在 NDEBUG 未定义时)。也可以通过定义 BOOST_CB_DISABLE_DEBUG 宏来显式关闭它。

更多的例子

以下例子包含了 circular_buffer 的各种用法。 

   #include <boost/circular_buffer.hpp>
   #include <numeric>
   #include <assert.h>

   int main(int /*argc*/, char* /*argv*/[])
   {
      // 创建一个容量为3的循环缓冲区
      boost::circular_buffer<int> cb(3);

      // 插入一些元素到循环缓冲区
      cb.push_back(1);
      cb.push_back(2);

      // 断言
      assert(cb[0] == 1);
      assert(cb[1] == 2);
      assert(!cb.full());
      assert(cb.size() == 2);
      assert(cb.capacity() == 3);

      // 再插入其它元素
      cb.push_back(3);
      cb.push_back(4);

      // 求和
      int sum = std::accumulate(cb.begin(), cb.end(), 0);

      // 断言
      assert(cb[0] == 2);
      assert(cb[1] == 3);
      assert(cb[2] == 4);
      assert(*cb.begin() == 2);
      assert(cb.front() == 2);
      assert(cb.back() == 4);
      assert(sum == 9);
      assert(cb.full());
      assert(cb.size() == 3);
      assert(cb.capacity() == 3);

      return 0;
   }

该 circular_buffer 的容量为3个 int. 因此，缓冲区的大小不会超过3。算法 accumulate 计算被存元素的总和。circular_buffer 的语义可以从断言进行推断。

有界缓冲区示例

有界缓冲区通常用于生产者-消费者模式，其中生产者线程产生一些事项并将它们保存在容器中，而消费者线程从容器中取出这些事项并处理它们。有界缓冲区必须保证当容器满时，生产者不会插入，同样当容器空时消费者不会取出，并且每个事项正好被一个消费者消耗。

下例示范了 circular_buffer 如何被用作有界缓冲区的底层容器。 

   #include <boost/circular_buffer.hpp>
   #include <boost/thread/mutex.hpp>
   #include <boost/thread/condition.hpp>
   #include <boost/thread/thread.hpp>
   #include <boost/progress.hpp>
   #include <boost/bind.hpp>

   template <class T>
   class bounded_buffer {
   public:

      typedef boost::circular_buffer<T> container_type;
      typedef typename container_type::size_type size_type;
      typedef typename container_type::value_type value_type;

      explicit bounded_buffer(size_type capacity) : m_unread(0), m_container(capacity) {}

      void push_front(const value_type& item) {
         boost::mutex::scoped_lock lock(m_mutex);
         m_not_full.wait(lock, boost::bind(&bounded_buffer<value_type>::is_not_full, this));
         m_container.push_front(item);
         ++m_unread;
         lock.unlock();
         m_not_empty.notify_one();
      }

      void pop_back(value_type* pItem) {
         boost::mutex::scoped_lock lock(m_mutex);
         m_not_empty.wait(lock, boost::bind(&bounded_buffer<value_type>::is_not_empty, this));
         *pItem = m_container[--m_unread];
         lock.unlock();
         m_not_full.notify_one();
      }

   private:
      bounded_buffer(const bounded_buffer&);              // 禁止复制构造函数
      bounded_buffer& operator = (const bounded_buffer&); // 禁止赋值操作符

      bool is_not_empty() const { return m_unread > 0; }
      bool is_not_full() const { return m_unread < m_container.capacity(); }

      size_type m_unread;
      container_type m_container;
      boost::mutex m_mutex;
      boost::condition m_not_empty;
      boost::condition m_not_full;
   };

bounded_buffer 使用了 Boost Threads 和 Boost Bind 库。

方法 push_front() 被生产者线程调用以插入新事项到缓冲区中。该方法锁住互斥体并等待至有空间容纳新事项。(互斥体在等待期间会被解锁，当条件满足时再重新锁住)。如果缓冲区中有空间可用，执行将继续，该方法插入新事项到 circular_buffer 的尾部。然后递增未读项计数并解锁互斥体(如果在互斥体解锁前抛出异常，互斥体会由 scoped_lock 的析构函数自动解锁)。最后该方法通知正在等待新事项被插入到缓冲区中的消费者线程。

方法 pop_back() 被消费者线程调用以从缓冲区读取一个事项。该方法锁住互斥体并等待至缓冲区中有未读项。如果有至少一个未读项，该方法就递减未读项计数并从 circular_buffer 读出下一个事项。然后解锁互斥体并通知正在等待缓冲区有空间容纳新事项的生产者线程。

方法 pop_back() 并不删除事项，被读事项保留在 circular_buffer 中，当 circular_buffer 满时它将被新事项(由生产者插入)所替换。该方法比显式调用 circular_buffer 的 pop_back() 方法删除该事项更为高效。这种方法基于以下假设，即新事项的赋值(替换)操作比析构(删除)旧项并原地构造(插入)新项更为高效。

要和基于不同容器的有界缓冲区进行比较，可以编译并运行 bounded_buffer_comparison.cpp. 该测试将表明，基于 circular_buffer 的有界缓冲区比基于 std::deque 的有界缓冲区更为高效。(在实践中的结果可能不一样，因为该测试会受到如即时CPU负荷等外部因素的影响)。

array_range array_one();

取得内部缓冲区的首个连续数组。

该方法与 array_two() 结合可用于将所存数据以数组方式传递给传统的 C API。假设有一个容量为10的 circular_buffer，包含7个字符 'a', 'b', ..., 'g'，其中 buff[0] == 'a', buff[1] == 'b', ... buff[6] == 'g':

circular_buffer<char> buff(10);

其内部的表示通常不是线性的，内部缓冲区的状态可能如下：

|e|f|g| | | |a|b|c|d|
end ---^
begin -------^

其中 |a|b|c|d| 表示"数组一"，|e|f|g| 表示"数组二"，而 | | | | 为空闲空间。
现在考虑一个典型的C风格函数，它写入数据到一个文件中：

int write(int file_desc, char* buff, int num_bytes);

有两种方法将 circular_buffer 中的内容写到一个文件中。要么使用 array_one() 和 array_two() 方法并调用 write 函数两次：

array_range ar = buff.array_one();
write(file_desc, ar.first, ar.second);
ar = buff.array_two();
write(file_desc, ar.first, ar.second);

或者使用 linearize() 方法：

write(file_desc, buff.linearize(), buff.size());

由于 array_one() 和 array_two() 方法的复杂度为常数，所以当 write 方法较为"廉价"时第一个选择更适合。另一方面，当调用 write 方法较为"昂贵"时第二个选择更为适合，linearize() 方法的调用是线性复杂度的。

返回：
内部缓冲区首个连续数组的数组区间。当 circular_buffer 为空时，返回数组的大小为 0.
抛出：
无。
异常安全性：
无抛出。
迭代器失效：
不会令任何迭代器失效。
复杂度：
常数(与 circular_buffer 的大小相关)。
警告：
通常，调用任何修改 circular_buffer 内部状态的方法都可能会破坏内部缓冲区的线性，并令得由 array_one() 和 array_two() (以及它们的常量版本)所返回的数组区间失效。
说明：
当内部缓冲区为线性时，如 |a|b|c|d|e|f|g| | | | ，"数组一"代表 |a|b|c|d|e|f|g| 而"数组二"不存在(array_two() 方法返回一个大小为 0 的数组)。
参见：
array_two(), linearize()

实际上只有4个元素被插入，而元素 3 和 4 被覆写。这是由于插入操作要保持容量不变。插入后的内部缓冲区如下：

|1|2|0|0|0|0|

作为比较，如果容量无需保持，则内部缓冲区会是这样: |1|2|0|0|0|0|0|3|4|.

将区间 [first, last) 插入到指定位置之前。
前提条件：
pos 为指向 circular_buffer 或其尾部的有效迭代器。 
有效区间 [first, last) 其中 first 和 last 符合 输入迭代器InputIterator 的要求。
作用:
区间 [first, last - max[0, distance(first, last) - (end() - pos) - reserve()]) 中的元素被插入到位置 pos 之前。
circular_buffer 尾部的 min[end() - pos, max[0, distance(first, last) - reserve()]] 个元素被覆写。
(详细解释请见例子)
参数：
pos
指定区间插入位置的迭代器。
first
被插入区间的开始。
last
被插入区间的结束。
抛出:
T::T(const T&) 抛出的任何异常。
T::operator = (const T&) 抛出的任何异常。
异常安全性:
基本；如果抛出一节中列出的操作没有抛出则不抛出异常。
迭代器失效:
指向插入点之前的迭代器(直至开始，除了 pos)均失效。指向被覆写元素的迭代器也失效。 
复杂度:
线性(与 [std::distance(begin(), pos) + std::distance(first, last)] 相关；如果 InputIterator 为 随机访问迭代器RandomAccessIterator 则与 min[capacity(), std::distance(begin(), pos) + std::distance(first, last)] 相关)。
例子：
考虑一个容量为6而大小为4的 circular_buffer。其内部缓冲区可能如下：

|1|2|3|4| | |
p ---^

在位置 p 之前插入一个区间的元素后：

insert(p, array, array + 5);

实际上只有指定区间中的元素 5, 6, 7 和 8 被插入，而元素 3 和 4 被覆写。这是由于插入操作要保持容量不变。插入后的内部缓冲区如下：

|1|2|5|6|7|8|

作为比较，如果容量无需保持，则内部缓冲区会是这样：|1|2|5|6|7|8|9|3|4|.

鸣谢

The circular_buffer has a short history. Its first version was a std::deque adaptor. This container was not very effective because of many reallocations when inserting/removing an element. Thomas Wenish did a review of this version and motivated me to create a circular buffer which allocates memory at once when created.

The second version adapted std::vector but it has been abandoned soon because of limited control over iterator invalidation.

The current version is a full-fledged STL compliant container. Pavel Vozenilek did a thorough review of this version and came with many good ideas and improvements. Also, I would like to thank Howard Hinnant, Nigel Stewart and everyone who participated at the formal review for valuable comments and ideas.
