对实数类型的概念性要求(Conceptual Requirements for Real Number Types)

这个库中的所有的函数和分布类型可以与满足下面给出的要求的任何的RealType 一起使用。所有的内建的(built-in)浮点类型都满足这些要求。用户定义的数据类型如果满足这些要求也可以与这个库一起使用。例如，一个对NTL (RR) 进行一个 廋包装类(thin wrapper class) 也可以使用。提交与其它的扩展精度类型进行绑定的代码也是非常受欢迎的!

在这些指导原则背后的要求是RealType 的行为与一个内建的(built in)的浮点类型的行为一样。

注意：

  1. 函数log_max_value 和函数 log_min_value 可以由其它函数合成，因此没有要求明确的特化(explicit specialisation)。
  2. 函数epsilon 可以由其它的函数合成，所以，如果RealType的精度在运行时不会发生改变，那么没有要求明确的特化(explicit specialisation)（参考boost/math/bindings/rr.hpp 头文件了解精度在运行时发生变化)。
  3. 函数digits, max_value 和 min_value，可以从std::numeric_limits自动合成。然而如果对类型RealType没有特化函数numeric_limits ，那么当代码试图使用这些函数的时候，你会得到一个编译错误，除非你明确地特化化这个函数。例如，如果 RealType的精度在运行时发生变化，那么对函数numeric_limits 的支持可能是不合适的。参考boost/math/bindings/rr.hpp 来查看一些例子。
提示

如果std::numeric_limits<> 没有针对RealType 类型进行特化，那么缺省的浮点精度为6个有效数字将会被其它包含的Boost程序所使用：

Boost.Test: 给出的迷惑性报错消息(misleading error messages)类似于

"difference between {9.79796} and {9.79796} exceeds 5.42101e-19%".

Boost.LexicalCast 和 Boost.Serialization 在将数值转换成字符串的时候会产生严重的精度损失。

虽然看起来应当要求 RealType 提供对std::numeric_limits 的特化，但是对于数字个数是一个运行时变量的 NTL::RR 和其它类似的类是不合理的(因为numeric_limits 是一个在编译时必须固定的值)。

注意，上面的表中只列举了那些已知会被这个库用到的标准库函数(或者有可能在将来的库的实现中用到)。下面的函数：acos, atan2, fmod, cosh, sinh, tanh, log10, lround, llround, ltrunc, lltrunc 以及modf` 当前并没有用到，但是如果将来增加新的特殊函数就可能用到。

注意，函数round, trunc 和 modf 并不是当前的C++标准的一部分：它们是C99标准的增加部分，并可能成为下一个C++标准的一部分。Boost库实现的这些函数作为一个备份 (backup)，这些函数总是称作未修饰的(unqualified)，以便可以进行参数依赖性查找( argument-dependent-lookup) 。

除此之外，对于高效和高精度的结果，非常期望有一个兰克泽斯逼近(Lanczos approximation) 。 你或许可以从 boost/math/special_functions/lanczos.hpp 或 boost/math/bindings/detail/big_lanczos.hpp 中修改一个已存在的逼近方法：对于前者，你需要修改 static_cast 为 lexical_cast ，并将常量改为字符串(为了确保系数不会被截断为 long double) 并且对于T类型特化lanczos_traits 。否则你需要研究libs/math/tools/lanczos_generator.cpp 来为你的RealType查找一个合适的逼近方法。如果你没有这样做，代码也可以编译，但是对于任何的调用 gamma/beta/erf 系列函数的函数，精度和效率都会大打折扣。
